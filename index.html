What the audit checks

Seven failure modes that actually leak tenant data in production, in order of blast radius:

  • RLS disabled — tables open to any role with grants (critical)
  • RLS on, zero policies — a latent gap about to be opened too wide (critical)
  • USING(true) policies — "RLS theatre": the checkbox passes, the policy enforces nothing (critical)
  • Write policies without WITH CHECK — read-isolated but a user can write into another tenant
  • Tables with no tenant column — un-scopable data (or unflagged global reference)
  • SECURITY DEFINER + mutable search_path — a privilege-escalation surface
  • anon write grants — defense-in-depth foot-gun
Honest about what a finding is. Each finding flags a pattern worth a human's eyes, not a guaranteed bug — a deliberate USING(true) on shared reference data is fine, for instance. The value is a triaged shortlist + exact remediation SQL + the judgment to tell a real leak from an intentional one. You can re-run the audit after fixes to confirm closure.
